# Nonull 智驾智能体 — 完整说明书

---

> **📌 重要声明 / Important Disclaimer**
>
> Nonull 是一个**内部使用的 ADAS 工程开发助手**，**不是**经过 ISO 26262 / ASIL-D 认证的车规级安全产品（非车规级、advisory only）。本项目中的"安全层"是**建议性（advisory）**的——它参考 ISO 26262 / MISRA / ASPICE 等标准的模式与术语进行风险提示和检查建议，但**并不实现** ASIL-D 要求的抗干扰、MC/DC 覆盖、形式化验证、SEooC 等条款。**请勿用于任何量产部署（非量产）、安全关键决策，或替代经过认证的安全机制。**

## 五、核心功能详解

### 1. 🤖 核心大脑

Nonull 不是简单的"输入-输出"机器。它会：

```
收到任务 → 思考怎么做 → 制定计划 → 执行动作（过安全关）→ 反思结果 → 输出
                                                         ↓
                                              没做完就接着做，做完了就结束
```

- 支持**同步/异步**两种方式
- 出错会自动重试（最多 3 次）
- 每次执行都记录到记忆里

### 2. 🧠 四种记忆

Nonull 有四种记忆，像人脑一样分工：

```
🧠 工作记忆    —— 记得当前正在做什么（短期）
📝 情景记忆    —— 记得之前做过什么任务（经历）
📚 知识记忆    —— 记得智驾领域的知识（法规、标准、算法）
🔧 技能记忆    —— 记得怎么操作（流程、步骤）
```

- 总容量可配置（默认 10K 条目/层，内存后端；如需更大规模可接 FAISS / Milvus / pgvector）
- 后台有"潜意识"在自动整理记忆
- 会的越多，做事越准

### 3. 🛡️ 安全卫士（五关检查）

每次执行动作前，都要过五关：

```
第1关：工具预过滤 —— 这个工具本身安全吗？
第2关：规则检查 —— 有 deny 规则禁止吗？（`safety_rules.yaml` 65 条规则 + `deny_first.py` 39 条默认规则）
第3关：风险评分 —— 风险多少分？（0-100分）
第4关：场景验证 —— 当前驾驶场景下安全吗？
第5关：事后验证 —— 执行结果有没有问题？
```

**任何一关不过，动作就不会执行。**

> ⚠️ **声明**：以上"安全卫士"是**建议性（advisory）**的，不实现 ISO 26262 ASIL-D 要求的抗干扰、MC/DC 覆盖、形式化验证、SEooC 等条款。请勿用于量产部署或安全关键决策。

内置的安全检查模板（基于标准模式，不是认证）：
- ✅ ISO 26262 模式（HARA / ASIL 风险提示）
- ✅ MISRA C:2012（代码规范建议）
- ✅ ASPICE（过程质量建议）

### 4. 🔄 多智能体协作

复杂任务可以拆成多个子任务，让多个"小 Nonull"一起干：

- 任务自动分解成 DAG（依赖关系图）
- 最多 8 个 Agent 并行工作
- 结果冲突时自动投票解决
- 8 个预置工作流（代码审查/安全分析/测试生成等）

### 5. 🔧 31 个专业技能

覆盖 9 个领域（实际计数，来自 `skills/__init__.py`）：

| 领域 | 技能数 | 做什么 |
|------|--------|-------|
| 代码 | 4 | 代码审查、代码优化、重构、Bug 检测 |
| 安全 | 4 | 危害分析、FMEA、ISO 26262 检查、安全案例 |
| 感知 | 4 | 传感器分析、感知模型审查、传感器标定、目标检测审查 |
| 规划 | 3 | 路径规划、行为规划、轨迹优化 |
| 测试 | 4 | 测试用例设计、SIL、HIL、回归测试 |
| 仿真 | 3 | 场景生成、CARLA 运行、边缘案例 |
| 数据 | 3 | 日志分析、数据管道审查、标注 QC |
| 研究 | 3 | 论文分析、SOTA 跟踪、算法对比 |
| DevOps | 3 | CI/CD、部署、监控 |

## 六、独有特色功能

下面这些是 **别的 AI 没有** 的功能：

### 👤 驾驶人格系统

Nonull 有三种"性格"，你可以按需切换：

| 人格 | 风格 | 适合场景 | 口头禅 |
|------|------|---------|--------|
| 🛡️ **保守派** | 谨慎、稳妥、注重冗余 | 安全分析、量产项目 | "这个场景的边界条件考虑了吗？" |
| 🚀 **运动派** | 激进、追求极限、创新 | 性能优化、算法评估 | "这个模块还有 30% 的优化空间！" |
| 🧓 **老司机** | 经验丰富、直觉锐利 | 场景分析、异常诊断 | "这种情况我见过，通常是..." |

**同一个代码，三种人格三种看法：**

```python
from persona import PersonaOrchestrator, PersonaType

# 保守派审查
agent = PersonaOrchestrator(PersonaType.CONSERVATIVE)
agent.analyze("这段自动变道代码安全吗？")
# → 关注：边界条件、冗余设计、失效模式

# 运动派审查  
agent = PersonaOrchestrator(PersonaType.SPORTY)
agent.analyze("这段代码性能还能提升吗？")
# → 关注：算法效率、延迟优化、并行计算

# 老司机审查
agent = PersonaOrchestrator(PersonaType.VETERAN)
agent.analyze("这个场景有没有见过类似问题？")
# → 关注：历史案例、经验教训、已知陷阱
```

### 🧠 场景思维（这个最厉害）

别的 AI 看到代码 → 分析语法。
**Nonull 看到代码 → 联想到驾驶场景。**

它内置了 **36 个标准驾驶场景**（覆盖 6 大域：高速/城市/泊车/天气/紧急/边缘）：

```
高速类：   高速巡航、前车切入、高速变道、...
城市类：   行人横穿、鬼探头、路口通行、...
泊车类：   垂直泊车、平行泊车、坡道起步、...
特殊类：   雨天行驶、夜间行驶、隧道通行、...
紧急类：   紧急制动、避让障碍、系统失效、...
```

当你让它审查代码时，它会自动想：
> "这段代码控制的场景有哪些？这些场景的边界测了吗？有没有漏掉的场景？"

```python
from persona import ScenarioEngine

engine = ScenarioEngine()

# 自动关联驾驶场景
report = engine.analyze_task_scenarios("审查 AEB 控制代码")
# 返回：这段代码涉及 [紧急制动、行人检测、雨天行驶] 等场景
# 建议补充 [夜间行人、鬼探头、湿滑路面] 等场景的测试

# 分析测试覆盖率
coverage = engine.analyze_scenario_coverage(existing_tests)
# 返回：覆盖率 65%，缺失 [高速 cut-in、隧道通行] 等场景
```

### 📊 安全指标记录

每一次安全操作都会记录到指标卡，advisory 建议性提示（非游戏化）：

```
🛡️ Safety Metric       —— 单次安全操作指标（0.0 - 1.0）
📈 Aggregate Level     —— 累计安全指标等级
⚠️ Risk Hint           —— 风险提示（非认证）
📊 Metric Card         —— 分类安全指标卡
```

```python
from persona import SafetyBadgeSystem

badges = SafetyBadgeSystem()
# 记录一次安全操作（advisory 指标）
badges.evaluate_interaction({
    "category": "AGGREGATE_SAFETY",
    "outcome": "success",
    "risk": 0.2,
})
print(badges.get_scorecard())
# → "Aggregate safety metric: 0.85 (level: GOOD)"
# → "总交互次数：1；平均分：0.85"
```

## 七、工作流模式

Nonull 有 8 个预设工作流，拿来就能用：

### 1. 代码审查工作流
```
提交代码 → 静态分析 → MISRA检查 → 安全审查 → 性能评估 → 报告生成
```

### 2. 安全分析工作流
```
系统定义 → 危害识别 → 风险评估 → ASIL定级 → 安全目标 → 安全案例
```

### 3. 测试生成工作流
```
需求分析 → 测试场景识别 → 用例生成 → 覆盖率检查 → 测试报告
```

### 4. 合规检查工作流
```
标准选择 → 逐项对照 → 差距分析 → 改进建议 → 合规报告
```

## 八、配置说明

### 简单配置

```yaml
# config/config.yaml
agent:
  mode: interactive            # interactive | autonomous | plan
  model:
    temperature: 0.2           # 越小越保守，越大越有创造性

memory:
  neocortex:
    enabled: true              # 开启记忆系统

safety:
  strictness: 4                # 安全严格度（1-5，5最严格）

persona:                       # 特色功能
  default: veteran             # 默认人格：老司机
  metrics: true                # 开启安全指标记录
```

### 环境变量配置

```bash
# LLM 配置（如果不设置，用本地默认）
export NONULL_LLM_PROVIDER=anthropic
export NONULL_LLM_API_KEY=sk-ant-xxxxx

# 安全等级
export NONULL_SAFETY_STRICTNESS=5

# 切换人格
export NONULL_PERSONA=sporty
```
